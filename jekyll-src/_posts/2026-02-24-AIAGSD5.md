## The Challenge: From Architectural Vision to Implementation

When humans build software, they rely on accumulated experience, team knowledge, and contextual understanding to translate architectural decisions into working code. AI agents lack this experiential knowledge. They can generate technically correct code that completely misses the architectural intent, leading to implementations that work in isolation but fail to form a coherent system.

Technology instruction files solve this by encoding architectural knowledge into consumable guidance that AI agents can follow consistently. These files serve as:

- **Pattern codebooks** that define implementation standards for specific architectural approaches
- **Constraint systems** that prevent AI agents from making decisions that violate architectural principles
- **Quality gates** that establish success criteria for generated code
- **Reference implementations** that provide concrete examples of correct patterns

The effectiveness of your AI-assisted development workflow directly correlates with the quality and completeness of your technology instruction files.

## Anatomy of a Technology Instruction File

Effective technology instruction files follow a structured approach that balances comprehensiveness with usability. They must be detailed enough to guide implementation decisions while remaining concise enough for AI agents to process efficiently within token limits.

The typical structure includes:

### 1. **Pattern Overview**
A brief explanation of the architectural pattern, its core principles, and when it should be applied. This provides context for all subsequent implementation guidance.

### 2. **Component Definitions**
Clear definitions of all key components, their responsibilities, and their relationships. This creates a shared vocabulary that ensures consistent implementation.

### 3. **Implementation Rules**
Specific, actionable directives written in imperative form. These rules constrain AI behavior and prevent common anti-patterns.

### 4. **Structural Templates**
Code templates and file organization patterns that establish the physical structure of implementations.

### 5. **Integration Guidance**
Instructions for how components interact with each other and with external systems, including dependency injection patterns and event flow.

### 6. **Testing Standards**
Specific testing approaches that validate both individual components and overall architectural compliance.

### 7. **Anti-Pattern Prevention**
Explicit guidance on what NOT to do, with explanations of why certain approaches violate the pattern.

## Case Study: CQRS Implementation Instructions

To illustrate these principles, I'll walk through creating technology instruction files for Command Query Responsibility Segregation (CQRS) with event sourcing. CQRS provides an excellent example because it's architecturally sophisticated yet commonly misimplemented by AI agents without proper guidance.

### Why CQRS as an Example?

CQRS separates read operations (queries) from write operations (commands), using distinct models for each. This pattern requires strict adherence to specific implementation rules:

- Commands must never return domain data
- Queries must never modify state
- Aggregates can only change state through event application
- Read models must be updated through projections, never directly

Without explicit instruction, AI agents frequently violate these principles, creating hybrid implementations that appear correct but break under load or lose consistency guarantees.

## Building Instruction Files: The Meta-Process

Creating effective technology instruction files requires a structured approach. Rather than writing these files manually, I use a meta-process that leverages AI to generate the instructions themselves.

### Step 1: Create a Prompt File

The first step involves creating a prompt file that will generate the instruction file. This might seem recursive, but it serves an important purpose: by having AI participate in creating the prompt, you get domain-specific context that improves the final instructions.

I created a [prompt-engineer.chatmode.md](https://github.com/johnmillerATcodemag-com/zeus.academia.3b/blob/a0ce78c1e527432044f574dafe4cca7cdacdd099/.github/chatmodes/prompt-engineer.chatmode.md) chat mode specifically for this task. The mode primes Copilot to think like a prompt engineer, focusing on generating prompts that produce high-quality instruction files.

The manual prompt I used: *"Create a prompt file that generates instruction files for a CQRS architecture. The prompt file should target AI agents and be optimized to reduce token consumption."*

### Step 2: Generate the Architecture Prompt

This produces a [generate-cqrs-eventsourcing-instructions.prompt.md](https://github.com/johnmillerATcodemag-com/zeus.academia.3b/blob/13fc7e8eb6fa47a5f933cd0e427dc7c65642f5c0/.github/prompts/generate-cqrs-eventsourcing-instructions.prompt.md) prompt file that accepts parameters for programming language, mediator framework, event store implementation, and projection strategy.

The generated prompt follows a rigid 15-section structure that ensures comprehensive coverage:
- Pattern overview and core concepts
- Reference tables for commands, events, aggregates, projections
- Implementation patterns with structural guidance
- Code templates for common scenarios
- Anti-pattern prevention
- Testing approaches
- Validation checklists

### Step 3: Execute the Prompt to Generate Instructions

Using the prompt: *"Submit the prompt generate-cqrs-eventsourcing-instructions.prompt.md in the C# language, the MediatR framework, the EventStoreDB event_store and inline projections"*

This generates a concrete instruction file that provides specific guidance for C# implementations using MediatR and EventStoreDB.

## Key Features of Effective Technology Instructions

The generated CQRS instruction file demonstrates several critical characteristics that make technology instruction files effective:

### 1. **Imperative Language**
Instructions use commanding language: "MUST implement," "NEVER allow," "ALWAYS use." This removes ambiguity and prevents AI agents from making judgment calls that could violate architectural principles.

### 2. **Explicit Anti-Patterns**
The file explicitly prohibits problematic patterns:
- Querying event stores for reads
- Mutating aggregate state outside event application
- Deleting events
- Sharing events between aggregates
- Loading multiple aggregates in command handlers

This preventive approach is crucial because AI agents excel at finding shortcuts that technically work but break architectural integrity.

### 3. **Structural Enforcement**
The instruction file mandates specific project structure:
```
Commands/
  ├── Handlers/
  └── Models/
Queries/
  ├── Handlers/
  └── Models/
Domain/
  ├── Aggregates/
  ├── Events/
  └── ValueObjects/
Infrastructure/
  ├── EventStore/
  ├── Projections/
  └── ReadModels/
```

This physical separation reinforces logical boundaries and prevents architectural drift.

### 4. **Technology-Specific Implementation**
Rather than generic guidance, the file provides concrete implementation details for the specific technology stack:
- MediatR's `IRequestHandler<TRequest, TResponse>` interface for all handlers
- EventStoreDB's `AppendAsync` with `ExpectedVersion` for optimistic concurrency
- C# records for immutable event types
- Specific naming patterns for event streams

### 5. **Validation Checkpoints**
The instruction includes validation criteria that AI agents can verify:
- Commands produce events without mutating state
- Queries access only read models
- Projections update synchronously with event appends
- Aggregates reconstruct state from event streams

## Implementation Impact: Before and After

To understand the value of technology instruction files, consider what happens without them:

### Without Instructions
AI agents generating CQRS implementations typically create:
- Commands that return data (violating command-query separation)
- Aggregates with public property setters (breaking immutability)
- Query handlers that access event stores directly (bypassing projections)
- Mixed concerns within single classes (violating separation)
- Inconsistent naming and structure across components

### With Instructions
The same AI agents, guided by technology instruction files, produce:
- Clean command-query separation with proper boundaries
- Immutable aggregates that change state only through events
- Read-only query handlers that access optimized projections
- Consistent structure and naming across all components
- Proper error handling and concurrency management

The difference is dramatic and measurable. Technology instruction files transform AI from a code generator into an architectural implementation partner.

## Best Practices for Technology Instruction Files

Based on implementing multiple technology instruction files, several patterns emerge:

### 1. **Start with Anti-Patterns**
Begin by documenting what NOT to do. AI agents are particularly prone to finding clever workarounds that break architectural principles. Explicitly prohibiting these patterns prevents architectural drift.

### 2. **Provide Code Templates**
Include minimal but complete code templates for common scenarios. These serve as both documentation and starting points for AI-generated code.

### 3. **Enforce Physical Structure**
Mandate specific folder structures and file naming conventions. Physical organization reinforces logical architecture and makes violations obvious.

### 4. **Include Testing Guidance**
Specify not just what to build, but how to test it. Include patterns for unit testing, integration testing, and architectural compliance testing.

### 5. **Version Your Instructions**
Technology instruction files evolve as you learn what works and what doesn't. Version them like code and maintain a changelog of architectural decisions.

### 6. **Validate with Real Implementation**
Test your instruction files by having AI agents implement realistic scenarios. Observe where the generated code deviates from your intent and refine the instructions accordingly.

## Governance Integration

Technology instruction files don't exist in isolation. They integrate with the broader governance framework established in earlier parts of this series:

- **Project instructions** define cross-cutting concerns and integration patterns
- **Technology instructions** provide implementation guidance for specific patterns
- **Git workflows** ensure proper review and validation of generated code
- **AI output policies** maintain audit trails and provenance tracking

This layered approach creates a comprehensive system that guides AI agents from initial requirements through final implementation.

## Context Validation

Before proceeding with implementation, I recommend running a context check to ensure your instruction files don't conflict with existing guidance. The `check-context.prompt.md` file examines current governance for:

- Conflicting instructions between files
- Factual inconsistencies in technical guidance
- Logical contradictions in architectural decisions
- Scope overlaps between different instruction files
- Technical incompatibilities between components
- Communication gaps in integration guidance
- Duplication and redundancy across files

This validation step becomes increasingly important as your instruction file library grows.

## Measuring Effectiveness

Technology instruction files should be treated as measurable assets. Track metrics like:

- **Architectural compliance rate**: Percentage of generated code that follows intended patterns
- **Consistency scores**: Similarity in structure and approach across components
- **Review cycle time**: How quickly generated code passes architectural review
- **Rework frequency**: How often generated code requires architectural corrections

These metrics help you refine your instruction files and demonstrate their value to stakeholders.
